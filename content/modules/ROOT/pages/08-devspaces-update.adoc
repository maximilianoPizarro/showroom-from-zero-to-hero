= Update Code with Dev Spaces

This module walks through **edit in web IDE → commit to Gitea → Tekton pipeline → new deployment**, using **Red Hat OpenShift Dev Spaces** and the **neuralbank-backend** component you created from templates.

== Open the component in Developer Hub

. Log in to **Developer Hub**.
. Find **neuralbank-backend** in the **Catalog** and open its page.

If the workshop enables the direct link:

* Click **Open in Dev Spaces** (or the equivalent plugin action).

If the button is missing:

* Copy the **repository URL** in Gitea from the component card.
* Open **Dev Spaces** in another tab and create or open a **Workspace** from that URL.

[.console-input]
----
Developer Hub -> Component neuralbank-backend -> Open in Dev Spaces
   (alternative) DevSpaces -> Add workspace -> Gitea repo URL
----

NOTE: The workspace should use the `devfile.yaml` generated by the template so you get the same tools (JDK, Maven, extensions) the pipeline expects.

== Wait for the workspace to start

. Wait until **DevWorkspace** reaches **Running** state.
. Open the IDE in the browser (VS Code–based or other per configuration).

If startup fails due to quota or permissions, read the message in the Dev Spaces console or ask the instructor for help.

== Make a code change

In the IDE file explorer, find a simple REST resource in the backend (for example a `*Resource.java` class under the API package).

Make a small, safe change, for example:

* Add a test `GET` endpoint that returns an identifiable message (`/api/hello-workshop`), **or**
* Change the response body of an existing endpoint to include a recognizable string (without breaking contracts the frontend uses).

Save the modified files.

WARNING: Avoid changing artifact names, Dockerfile, or pipeline paths unless the workshop asks; small code or message changes are enough to demonstrate CI/CD.

== Build locally in the workspace

Open a terminal in Dev Spaces and run the Maven package for the project:

[.console-input]
----
cd /projects/neuralbank-backend   # adjust the path if the clone uses a different name
mvn -q package -DskipTests
----

Fix compile errors before continuing; the remote pipeline will fail the same way.

NOTE: If `mvn` is not on PATH, use the IDE **Maven** task or the container defined in the `devfile` per environment documentation.

== Commit and push to Gitea

From the terminal or the IDE source control view:

[.console-input]
----
git status
git add <modified-files>
git commit -m "workshop: demo change from Dev Spaces"
git push origin <main-branch>   # main or master depending on the repo
----

If Git asks for credentials, use the workshop mechanism (token, SSO, or credentials embedded in Dev Spaces).

== Watch the Tekton pipeline

. Go to **OpenShift Console -> Pipelines -> PipelineRuns** in project `{user_name}-neuralbank`.
. Identify a new run triggered by the recent **push**.

Open task logs (`git-clone`, `maven-build`, `build-image`, `deploy`) and confirm the new image is built and deployed.

NOTE: Some environments use explicit webhooks; others use polling. If the pipeline does not start within a few minutes, check events in Gitea and trigger configuration.

== Verify deployment of the new version

. Under **Workloads -> Pods**, confirm the Deployment performed a **rollout** (new pods with a recent timestamp).
. Optional: inspect **image digest** or tag on the Deployment to confirm it points to the new build.

Try the endpoints again on the public **Route**:

[.console-input]
----
curl -sk "https://<route-host>/api/<your-modified-endpoint-or-resource>"
----

You should see the change reflected (new JSON, different message, etc.).

== Closing the loop

You demonstrated the full cycle **Dev Spaces → Git → Tekton → OpenShift**, aligned with **GitOps** and the component card in **Developer Hub**. This is the working model organizations want when adopting Developer Hub: developers stay in a consistent environment while the platform applies standards and automation.
