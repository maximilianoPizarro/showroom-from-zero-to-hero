= Explore Pipelines and Topology

After deploying **neuralbank-backend**, it is useful to inspect how **Tekton** materialized CI/CD and how **OpenShift** represents the application in the topology view. This module guides that exploration and links to the **Software Catalog** in Developer Hub.

== Open the Pipelines console

. Log in to the **OpenShift Console**.
. Select project **`{user_name}-neuralbank`**.
. In the sidebar, go to **Pipelines** (or **CI/CD -> Pipelines** depending on console version).

Here you will see the **Pipeline** definition and recent **PipelineRuns** generated when creating or updating the repository.

== Locate the neuralbank-backend PipelineRun

. On the **PipelineRuns** tab, sort by **Started** (most recent first).
. Identify a run whose name or labels reference `neuralbank-backend` or the Gitea repository.

NOTE: If you see no runs, confirm that the trigger or initial PipelineRun was created from the template; a push to the repo may be required to fire the pipeline depending on the workshop configuration.

== Review pipeline tasks

Open the **PipelineRun** and examine the list of **Tasks** in order. A typical workshop flow includes:

[cols="1,2",options="header"]
|===
|Task |What to look for

|`git-clone`
|Repository and revision cloned; check parameters (Gitea URL, branch).

|`maven-build`
|Java build; `mvn` logs and generated artifact.

|`build-image`
|Container image build (Buildah/Kaniko or another tool per template).

|`deploy`
|Deployment update on OpenShift; verify the new image is referenced on the resource.
|===

For each task, open the **logs** and confirm it ended in **Succeeded**. If any fail, the message usually points to credentials, resource limits, or build errors.

== Use the Topology view

. In the console, open the **Topology** view for project `{user_name}-neuralbank`.
. Locate the backend **Deployment** (circular icon with status ring).
. Click the node to open the side panel: **Pods**, **Services**, **Routes**, and recent events.

Topology shows visual relationships between components—for example, how the **Service** routes traffic to the Deployment **Pods**.

== Explore application details

From the component panel in Topology or dedicated menus:

* **Pods**: replica count, `Running` state, restarts, and quick metrics.
* **Services**: internal ports and selectors pointing to the Deployment.
* **Routes**: public or cluster URL to try the API (consistent with the deployment module).

NOTE: If the workshop adds **HorizontalPodAutoscaler** or other operators, additional resources may appear linked to the same Deployment.

== Return to Developer Hub

. Open **Developer Hub**.
. Locate component **neuralbank-backend** in the catalog.
. From the entity card, check links to **source** (Gitea), documentation, and any **CI/CD** annotation exposed by the pipelines or Git plugin.

This connects the “infrastructure” view (OpenShift/Tekton) with the “product” view (catalog and ownership in the Hub).

== Summary

You followed a **PipelineRun** task by task, validated deployment state in **Topology** for namespace `{user_name}-neuralbank`, and related the component to its entry in **Developer Hub**. This completes the operational picture of the Neuralbank backend before adding frontend and MCP.
