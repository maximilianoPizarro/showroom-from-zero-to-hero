= Arquitectura general del workshop

Esta página describe la arquitectura lógica del escenario Neuralbank: cómo encajan Red Hat Developer Hub, GitOps, CI/CD, identidad y exposición segura de APIs frente a un clúster OpenShift.

== Vista de componentes

El siguiente diagrama muestra los componentes principales del workshop y cómo interactúan entre sí:

image::02-component-view.svg[Diagrama de componentes del workshop, width=100%, link=_images/02-component-view.svg]

Cada bloque tiene un rol claro: el Hub orquesta la experiencia; Git es la fuente de verdad; Argo CD aplica el modelo declarativo; Tekton automatiza la cadena de suministro de software; la capa de gateway y políticas expone y protege el tráfico; Dev Spaces acerca el desarrollo al entorno real del clúster.

== Flujo principal: de la plantilla al despliegue

El recorrido típico que seguirás en el workshop es:

image::02-scaffolding-flow.svg[Flujo de scaffolding desde Developer Hub hasta el despliegue, width=100%, link=_images/02-scaffolding-flow.svg]

. El desarrollador abre **Developer Hub** y lanza una **Software Template** (por ejemplo Neuralbank Backend).
. La plantilla **genera o publica** un repositorio en **Gitea** con código, manifiestos, `devfile` y definición de pipeline.
. **Argo CD** detecta la aplicación (Application) asociada y **sincroniza** manifiestos hacia el namespace `{user_name}-neuralbank` en OpenShift.
. **Tekton** ejecuta un **PipelineRun** (clonado, build Maven, build de imagen, despliegue) según los recursos creados en el repo.
. OpenShift **programa los pods**, expone **Services** y, según el caso, recursos de **Gateway API** y rutas HTTP.
. El componente queda **registrado en el catálogo** de Developer Hub para descubrimiento, documentación y relaciones (por ejemplo APIs).

Este patrón une **golden path** (plantilla) con **GitOps** (Argo CD) y **CI/CD** (Tekton), manteniendo trazabilidad desde el primer clic en el Hub hasta el pod en ejecución.

== Patrón Connectivity Link (Gateway + rutas + políticas)

Para exponer servicios hacia el exterior o entre equipos con controles uniformes, el escenario utiliza un patrón alineado con **Gateway API** e integración con **Istio** y **Kuadrant** para políticas:

image::02-connectivity-link.svg[Patrón Connectivity Link con Gateway API e Istio, width=100%, link=_images/02-connectivity-link.svg]

* **Gateway**: define el punto de entrada del tráfico (host, listeners, TLS según configuración).
* **HTTPRoute**: enlaza hostnames y reglas de enrutamiento con los Services backend.
* **OIDCPolicy**: exige autenticación OIDC coherente con tu IdP (p. ej. Keycloak), protegiendo el tráfico de la API expuesta.
* **RateLimitPolicy**: aplica límites de tasa para proteger backends y cumplir políticas de uso justo o defensa ante abuso.

En conjunto, estas piezas forman un **connectivity link** declarativo: el mismo enfoque GitOps que el resto del sistema, revisable en Git y aplicado por operadores en el clúster.

|Model Context Protocol — protocolo para servidores de herramientas de IA

|OpenShift Lightspeed — asistente AI integrado en OpenShift

|Gestión declarativa de infraestructura y aplicaciones desde Git

|Horizontal Pod Autoscaler — escalado automático de pods

|Transport Layer Security — cifrado de comunicaciones

|SPA

|LLM
|Large Language Model — modelo de lenguaje grande

== Rol de cada componente

[cols="1,3",options="header"]
|===
|Componente |Rol

|Developer Hub
|Portal del desarrollador: catálogo, plantillas, documentación y plugins hacia GitOps, pipelines y entornos.

|Gitea
|Repositorio Git interno: código fuente, manifiestos y triggers para pipelines.

|Argo CD
|Sincronización continua desde Git al estado del clúster; salud y drift visibles en el dashboard de Argo CD.

|Tekton
|Ejecución de pipelines como recursos de Kubernetes; encadena tareas de CI/CD.

|Keycloak
|Identidad y SSO; alimenta políticas OIDC y acceso al Hub.

|Dev Spaces
|Entornos de desarrollo basados en `devfile`, conectados al mismo repo que GitOps y Tekton.

|Gateway API / Istio / Kuadrant
|Entrada norte-sur del tráfico, enrutamiento y políticas (OIDC, rate limit) sobre las APIs expuestas.

|OpenShift Lightspeed (OLS)
|Asistente AI integrado con MCP Gateway para operaciones de clúster vía herramientas.

|LiteLLM
|Proxy de modelos LLM; gestiona Virtual Keys, rate limits y routing entre proveedores de AI.
|===

== Lectura para el workshop

Durante los módulos posteriores volverás a este mapa mental: cada vez que crees una plantilla, mira el repo en Gitea; cada vez que sincronice Argo CD, revisa el namespace en OpenShift; cuando el pipeline termine, valida imagen y despliegue; cuando expongas el MCP o APIs, relaciona Gateway, HTTPRoute y políticas con lo que ves en consola y en el catálogo.

Con esta arquitectura, **Developer Hub** actúa como fachada humana sobre un sistema declarativo y automatizado que lleva el software desde el repositorio hasta producción de forma repetible.
