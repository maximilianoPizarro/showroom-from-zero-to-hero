= Workshop Architecture Overview

This page describes the logical architecture of the Neuralbank scenario: how Red Hat Developer Hub, GitOps, CI/CD, identity, and secure API exposure fit together in front of an OpenShift cluster.

== Component View

The following diagram shows the main workshop components and how they interact:

image::02-component-view.svg[Workshop component diagram, width=100%, link=_images/02-component-view.svg]

Each block has a clear role: the Hub orchestrates the experience; Git is the source of truth; Argo CD applies the declarative model; Tekton automates the software supply chain; the gateway and policy layer exposes and protects traffic; Dev Spaces brings development close to the cluster’s real environment.

== Main Flow: From Template to Deployment

The typical path you will follow in the workshop is:

image::02-scaffolding-flow.svg[Scaffolding flow from Developer Hub to deployment, width=100%, link=_images/02-scaffolding-flow.svg]

. The developer opens **Developer Hub** and runs a **Software Template** (for example Neuralbank Backend).
. The template **generates or publishes** a repository in **Gitea** with code, manifests, `devfile`, and pipeline definition.
. **Argo CD** detects the associated Application and **synchronizes** manifests to the `{user_name}-neuralbank` namespace on OpenShift.
. **Tekton** runs a **PipelineRun** (clone, Maven build, image build, deployment) according to the resources created in the repo.
. OpenShift **schedules pods**, exposes **Services**, and, depending on the case, **Gateway API** resources and HTTP routes.
. The component is **registered in the Developer Hub catalog** for discovery, documentation, and relationships (for example APIs).

This pattern joins **golden path** (template) with **GitOps** (Argo CD) and **CI/CD** (Tekton), keeping traceability from the first click in the Hub to the running pod.

== Connectivity Link Pattern (Gateway + routes + policies)

To expose services externally or between teams with uniform controls, the scenario uses a pattern aligned with **Gateway API** and integration with **Istio** and **Kuadrant** for policies:

image::02-connectivity-link.svg[Connectivity Link pattern with Gateway API and Istio, width=100%, link=_images/02-connectivity-link.svg]

* **Gateway**: defines the traffic entry point (host, listeners, TLS per configuration).
* **HTTPRoute**: binds hostnames and routing rules to backend Services.
* **OIDCPolicy**: requires OIDC authentication consistent with your IdP (e.g. Keycloak), protecting exposed API traffic.
* **RateLimitPolicy**: applies rate limits to protect backends and meet fair-use or abuse-defense policies.

Together, these pieces form a declarative **connectivity link**: the same GitOps approach as the rest of the system, reviewable in Git and applied by operators on the cluster.

== Role of Each Component

[cols="1,3",options="header"]
|===
|Component |Role

|Developer Hub
|Developer portal: catalog, templates, documentation, and plugins toward GitOps, pipelines, and environments.

|Gitea
|Internal Git repository: source code, manifests, and pipeline triggers.

|Argo CD
|Continuous synchronization from Git to cluster state; health and drift visible in the Argo CD dashboard.

|Tekton
|Pipeline execution as Kubernetes resources; chains CI/CD tasks.

|Keycloak
|Identity and SSO; feeds OIDC policies and Hub access.

|Dev Spaces
|Development environments based on `devfile`, connected to the same repo as GitOps and Tekton.

|Gateway API / Istio / Kuadrant
|North-south traffic ingress, routing, and policies (OIDC, rate limit) on exposed APIs.

|OpenShift Lightspeed (OLS)
|AI assistant integrated with MCP Gateway for cluster operations via tools.

|LiteLLM
|LLM model proxy; manages Virtual Keys, rate limits, and routing between AI providers.
|===

== Reading for the Workshop

In later modules you will return to this mental map: each time you create a template, look at the repo in Gitea; each time Argo CD syncs, check the namespace in OpenShift; when the pipeline finishes, validate image and deployment; when you expose the MCP or APIs, relate Gateway, HTTPRoute, and policies to what you see in the console and in the catalog.

With this architecture, **Developer Hub** acts as the human-facing layer over a declarative, automated system that takes software from repository to production in a repeatable way.
