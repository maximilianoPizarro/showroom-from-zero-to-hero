= Explore Connectivity Link: OIDC Policy (Neuralbank)

In this module you will explore how **Red Hat Connectivity Link** protects the pre-deployed **Neuralbank** stack using **OIDCPolicy** (OpenID Connect via Keycloak) and **RateLimitPolicy**.

== Architecture Overview

The Neuralbank stack in the `neuralbank-stack` namespace consists of:

[cols="1,2"]
|===
| Component | Description

| `neuralbank-backend`
| Quarkus REST API exposing `/api/v1/customers`

| `neuralbank-db`
| PostgreSQL database with seed data

| `neuralbank-frontend`
| Web interface for credit visualization

| `neuralbank-gateway-istio`
| Istio ingress gateway (auto-managed by Gateway API)
|===

The request flow is:

----
Client -> Istio Gateway -> OIDCPolicy (Keycloak JWT validation) -> RateLimitPolicy (60 req/min) -> HTTPRoute -> Backend Service
----

== Inspect the Gateway and HTTPRoute

[.console-input]
[source,bash]
----
oc get gateway -n neuralbank-stack
oc get httproute -n neuralbank-stack
----

The Gateway should show `PROGRAMMED: True`. The HTTPRoute forwards `/api` and `/q` paths to the backend.

== Verify the OIDCPolicy

[.console-input]
[source,bash]
----
oc get oidcpolicy -n neuralbank-stack -o wide
----

The OIDCPolicy links to the Keycloak realm `neuralbank` and protects the HTTPRoute. It requires a valid JWT Bearer token in the `Authorization` header.

Key configuration:

* **Issuer**: Keycloak realm `neuralbank`
* **Protected paths**: `/api/*`
* **Unauthorized behavior**: HTTP 302 redirect to Keycloak login

== Verify the RateLimitPolicy

[.console-input]
[source,bash]
----
oc get ratelimitpolicy -n neuralbank-stack -o wide
----

The RateLimitPolicy caps traffic at **60 requests per minute** per authenticated user.

== Test the API

Without a token, the gateway redirects to Keycloak login (HTTP 302):

[.console-input]
[source,bash]
----
curl -s -o /dev/null -w "HTTP %{http_code}\n" \
  https://neuralbank.apps.cluster-l9nhj.dynamic.redhatworkshops.io/api/v1/customers
----

Expected output:

[.console-output]
----
HTTP 302
----

This confirms the OIDCPolicy is enforcing authentication.

== How OIDC Authentication Works

. The client sends a request without a token
. The OIDCPolicy intercepts and returns HTTP 302 to Keycloak's authorization endpoint
. The user authenticates in Keycloak (username/password)
. Keycloak issues a JWT token and redirects back
. Subsequent requests include `Authorization: Bearer <token>`
. The OIDCPolicy validates the JWT signature and claims
. If valid, the request reaches the backend

== Summary

The OIDC pattern is ideal for **interactive user-facing applications** where:

* Users authenticate through a web browser
* Tokens are issued by an identity provider (Keycloak)
* Fine-grained user identity is needed for authorization
* Rate limiting can be applied per authenticated user
