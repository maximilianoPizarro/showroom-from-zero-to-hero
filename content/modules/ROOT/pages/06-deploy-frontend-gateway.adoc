= Desplegar frontend y gateway

En este módulo desplegarás el **frontend** Neuralbank y el servicio **Customer Service MCP** mediante Software Templates, y comprobarás cómo el escenario MCP incorpora objetos de **Gateway API** y políticas de conectividad. Todos los servicios se desplegarán en tu namespace `{user_name}-neuralbank`.

== Contexto

Ya deberías tener **neuralbank-backend** en ejecución en el namespace `{user_name}-neuralbank`. Ahora añadirás:

* **neuralbank-frontend**: interfaz web para visualizar datos de créditos expuestos por el backend.
* **customer-service-mcp**: servidor MCP con recursos adicionales para el patrón de **connectivity link** (Gateway, HTTPRoute, OIDCPolicy, RateLimitPolicy).

== Paso 1: Desplegar Neuralbank Frontend

. En **Developer Hub**, ve a **Create** y elige la plantilla **Neuralbank: Frontend**.
. Completa el formulario:
  * **Nombre**: `neuralbank-frontend`
  * **Usuario**: `{user_name}` (se autocompleta si estás logueado; esto desplegará en `{user_name}-neuralbank`)
. Ejecuta **Create** y espera a que finalicen los pasos del scaffolder.

[.console-input]
----
Developer Hub -> Create -> "Neuralbank: Frontend"
name:  neuralbank-frontend
owner: {user_name}
→ namespace: {user_name}-neuralbank
----

NOTE: Si el frontend espera una URL de API configurable, comprueba en el repo generado (Gitea) variables de entorno o `ConfigMap` que apunten al backend; ajústalas solo si el taller lo indica.

== Paso 2: Desplegar Customer Service MCP

. Repite el flujo **Create** con la plantilla **Neuralbank: Customer Service MCP**.
. Usa un nombre claro: `customer-service-mcp`.
. Mantén el **owner** como `{user_name}` para que se despliegue en el mismo namespace.

[.console-input]
----
Developer Hub -> Create -> "Neuralbank: Customer Service MCP"
name:  customer-service-mcp
owner: {user_name}
→ namespace: {user_name}-neuralbank
----

Esta plantilla no solo genera código: materializa el patrón de **connectivity link** con objetos como **Gateway**, **HTTPRoute**, **OIDCPolicy** y **RateLimitPolicy**.

== Paso 3: Verificar repositorios en Gitea

En **Gitea** (`https://gitea.{cluster_domain}`), confirma que existen dos repositorios nuevos bajo tu usuario:

* `{user_name}/neuralbank-frontend` — Dockerfile, assets estáticos, manifiestos, `devfile`.
* `{user_name}/customer-service-mcp` — fuentes Java, manifiestos y recursos de gateway/políticas.

Revisa que los **PipelineRuns** se hayan disparado.

== Paso 4: Comprobar aplicaciones en Argo CD

. Abre **Argo CD**: `https://openshift-gitops-server-openshift-gitops.{cluster_domain}`
. Localiza **Applications** para frontend y MCP en el namespace `{user_name}-neuralbank`.
. Verifica **sync** y **health**; sincroniza manualmente si tu rol lo permite.

NOTE: Los recursos de Gateway API y políticas pueden aparecer como objetos adicionales en el grafo de la aplicación MCP; si alguno está en estado degradado, revisa eventos en OpenShift y la configuración del operador (Kuadrant / Istio).

== Paso 5: Validar los tres componentes en el catálogo

. Entra en **Developer Hub -> Catalog**.
. Filtra por owner `{user_name}` o busca por nombre.
. Deberías ver **tres** componentes: `neuralbank-backend`, `neuralbank-frontend` y `customer-service-mcp`.

Abre cada ficha y comprueba enlaces al código, documentación y relaciones (dependencias hacia APIs).

== Paso 6: Inspeccionar Gateway y HTTPRoute en OpenShift

. En la **OpenShift Console**, selecciona el proyecto `{user_name}-neuralbank`.
. Navega a **Networking** y busca recursos de **Gateway** y **HTTPRoute**.
. Anota el **hostname** o estado de los listeners y las reglas que enrutan hacia el Service del MCP.

[.console-input]
----
OpenShift Console -> Project: {user_name}-neuralbank -> Gateway / HTTPRoute -> host y backends
----

Opcionalmente, revisa **OIDCPolicy** y **RateLimitPolicy** asociados para entender cómo se protege el endpoint expuesto.

== Paso 7: Probar el frontend

Abre la **Route** del frontend en el navegador. Comprueba que la interfaz carga y consume datos del backend.

WARNING: Errores CORS o de URL de API suelen deberse a configuración del frontend o a rutas internas; revisa variables en el Deployment antes de cambiar políticas de red.

== Resumen

Has añadido **frontend** y **MCP** con plantillas en el namespace `{user_name}-neuralbank`, verificaste **GitOps** y **catálogo**, y localizaste **Gateway** y **HTTPRoute** que formalizan la conectividad segura del MCP. El siguiente módulo profundiza en **Argo CD**, entidades de **API** y políticas **Kuadrant**.
