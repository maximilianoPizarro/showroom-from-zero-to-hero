= Deploy Frontend and Gateway

In this module you will deploy the Neuralbank **frontend** and the **Customer Service MCP** service using Software Templates, and you will see how the MCP scenario includes **Gateway API** objects and connectivity policies. All services deploy in your `{user_name}-neuralbank` namespace.

== Context

You should already have **neuralbank-backend** running in namespace `{user_name}-neuralbank`. You will now add:

* **neuralbank-frontend**: web UI to view credit data exposed by the backend.
* **customer-service-mcp**: MCP server with extra resources for the **connectivity link** pattern (Gateway, HTTPRoute, OIDCPolicy, RateLimitPolicy).

== Step 1: Deploy Neuralbank Frontend

. In **Developer Hub**, go to **Create** and choose template **Neuralbank: Frontend**.
. Complete the form:
  * **Name**: `neuralbank-frontend`
  * **User**: `{user_name}` (autocompletes when logged in; deploys to `{user_name}-neuralbank`)
. Run **Create** and wait until scaffolder steps finish.

[.console-input]
----
Developer Hub -> Create -> "Neuralbank: Frontend"
name:  neuralbank-frontend
owner: {user_name}
→ namespace: {user_name}-neuralbank
----

NOTE: If the frontend expects a configurable API URL, check environment variables or `ConfigMap` in the generated repo (Gitea) that point to the backend; adjust only if the workshop says so.

== Step 2: Deploy Customer Service MCP

. Repeat **Create** with template **Neuralbank: Customer Service MCP**.
. Use a clear name: `customer-service-mcp`.
. Keep **owner** as `{user_name}` so it deploys in the same namespace.

[.console-input]
----
Developer Hub -> Create -> "Neuralbank: Customer Service MCP"
name:  customer-service-mcp
owner: {user_name}
→ namespace: {user_name}-neuralbank
----

This template does not only generate code: it materializes the **connectivity link** pattern with objects such as **Gateway**, **HTTPRoute**, **OIDCPolicy**, and **RateLimitPolicy**.

== Step 3: Verify repositories in Gitea

In **Gitea** (`https://gitea-gitea.{cluster_domain}`), confirm two new repositories exist under your user:

* `{user_name}/neuralbank-frontend` — Dockerfile, static assets, manifests, `devfile`.
* `{user_name}/customer-service-mcp` — Java sources, manifests, and gateway/policy resources.

Check that **PipelineRuns** have been triggered.

== Step 4: Check applications in Argo CD

. Open **Argo CD**: `https://openshift-gitops-server-openshift-gitops.{cluster_domain}`
. Locate **Applications** for frontend and MCP in namespace `{user_name}-neuralbank`.
. Verify **sync** and **health**; sync manually if your role allows.

NOTE: Gateway API resources and policies may show as extra objects on the MCP application graph; if any are degraded, review events in OpenShift and operator configuration (Kuadrant / Istio).

== Step 5: Validate the three components in the catalog

. Open **Developer Hub -> Catalog**.
. Filter by owner `{user_name}` or search by name.
. You should see **three** components: `neuralbank-backend`, `neuralbank-frontend`, and `customer-service-mcp`.

Open each card and check links to code, documentation, and relationships (dependencies on APIs).

== Step 6: Inspect Gateway and HTTPRoute in OpenShift

. In **OpenShift Console**, select project `{user_name}-neuralbank`.
. Go to **Networking** and find **Gateway** and **HTTPRoute** resources.
. Note the **hostname** or listener state and rules that route to the MCP Service.

[.console-input]
----
OpenShift Console -> Project: {user_name}-neuralbank -> Gateway / HTTPRoute -> host and backends
----

Optionally review associated **OIDCPolicy** and **RateLimitPolicy** to see how the exposed endpoint is protected.

== Step 7: Try the frontend

Open the frontend **Route** in the browser. Confirm the UI loads and consumes data from the backend.

WARNING: CORS or API URL errors usually come from frontend configuration or internal routes; check variables on the Deployment before changing network policies.

== Summary

You added **frontend** and **MCP** with templates in namespace `{user_name}-neuralbank`, verified **GitOps** and the **catalog**, and located **Gateway** and **HTTPRoute** that define secure MCP connectivity.
