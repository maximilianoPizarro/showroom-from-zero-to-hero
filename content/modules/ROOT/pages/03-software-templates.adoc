= Understanding Software Templates

**Software Templates** are one of the pillars of Backstage and Red Hat Developer Hub. They let the organization package opinions, standards, and automation into flows the developer runs through a secure form.

== What Are Software Templates?

A Software Template is a declarative definition that describes:

* **What** will be generated (code, manifests, pipelines, catalog-info, etc.).
* **What parameters** the user must provide (component name, owner, visibility).
* **What steps** the template engine runs in sequence (substitute files, publish to Git, register in the catalog).

In practice, they turn an internal multi-hour procedure into a minutes-long flow with traceability and compliance.

== Typical Structure: `template.yaml` and `skeleton/`

In a templates repository you will usually find:

* **`template.yaml`** (or equivalent): template metadata (`apiVersion`, `kind: Template`), `parameters` and `steps` sections. This is where the Hub-visible name, form fields, and actions are declared.
* **`skeleton/` directory**: tree of template files with **placeholders** the engine replaces with parameter values (for example project name, group, image paths).

The `skeleton` directory may contain:

* Source code (Java, Node, etc.).
* Kubernetes/OpenShift manifests (`deployment`, `service`, `route`, Gateway API resources).
* `devfile.yaml` for **Dev Spaces**.
* `tekton/pipeline.yaml` or other CI/CD resources.
* `catalog-info.yaml` for the **Backstage Catalog**.

== Common Parameters and Steps

**Parameters** typically include:

* Component or repository name.
* Owner (`owner`) or team, aligned with the catalog entity model.
* Sometimes cluster URL, visibility, or tags—always bounded by whoever maintains the template.

Frequent **steps** in GitOps ecosystems are:

* **`fetch:template`** (or equivalent): copies the `skeleton` and applies substitutions from parameters.
* **`publish:gitea`** / **`publish:github`** / similar: creates or updates the remote repository with the generated content.
* **`catalog:register`**: registers the component (and sometimes the API) in the Software Catalog so it appears in Developer Hub with links and metadata.

The exact naming depends on the **scaffolder actions** installed in your instance; what matters is recognizing the pattern: *generate → publish to Git → register*.

== Neuralbank Templates Available in the Workshop

For the Neuralbank case you will work with three templates designed to fit together:

|**MCP** (Model Context Protocol) server for customer support; includes exposure pieces and policies associated with the connectivity pattern.

Each implements a distinct but coherent **golden path** for the same business domain.

== What Does Each Template Generate?

In general, when you run a Neuralbank template you will get:

* **Code** ready to compile and extend (sources, `pom.xml` or equivalent, Dockerfile if applicable).
* **OpenShift deployment manifests** (Deployment, Service, Route or routing objects as appropriate).
* **Tekton pipeline definition** aligned with the repo (for image build and deployment).
* **`devfile.yaml`** to open the project in **Red Hat OpenShift Dev Spaces** with preconfigured tools and commands.
* **Connectivity link objects** where applicable—especially in the **MCP** scenario—: resources such as **Gateway**, **HTTPRoute**, **OIDCPolicy**, and **RateLimitPolicy**, to expose the service with OIDC authentication and rate limits.

In addition, **`catalog-info.yaml`** (or another registration mechanism) lets Developer Hub show the component with **owner**, **system**, and relationships to APIs and dependencies.

== Best Practices When Using Templates

* Review **default values** and each parameter’s **description** before creating the component.
* After creation, open the **repository in Gitea** and confirm the structure matches expectations.
* Find the **Component** entity (and **API** if applicable) in the catalog to link documentation and pipelines.

Mastering Software Templates means mastering the **contract** between development and platform: you supply business context in the form; the platform applies the rest in a repeatable way.
